# Feedback Loop Analysis: 2026-02-12 (Session 9)

## Ground Truth Status
- Resolved forecasts with our predictions: 0 (Metaculus API returns `resolution: null` for all resolved questions)
- Live forecasts: 95
- Retrodictions: ~40 (17 binary, 10 numeric, rest incomplete)
- Mean binary retrodiction Brier: 0.2558 (at naive baseline — no forecasting skill demonstrated)

### Critical Finding: Resolution Data Unavailable
The Metaculus API does not expose `question.resolution` values. All resolved questions return `resolution: null` from both the JSON API and the DRF HTML fallback. This blocks calibration analysis on live forecasts entirely. Only retrodictions (where we know resolution from the question text) provide calibration data.

**Impact**: Cannot evaluate live forecast accuracy. All calibration must come from retrodictions.

## Object-Level Findings

### Systematic Hedging Problem
44% of binary forecasts fall in the 40-60% zone. Stock questions always 0.47-0.55. The agent rarely takes strong directional positions.

**Root causes identified:**
1. **Prompt framing**: "Nothing Ever Happens" section had prescriptive logit adjustments (-0.5 to -1.5 logits) that normalized toward 50% rather than toward low probability
2. **Meta-prediction guidance**: "Key Uncertainty" section told the agent "predicting forecaster behavior is hard" — priming it to default to 50%
3. **Subagents never used**: 0/52 sessions used subagents. The prompt said "Most forecasting questions are simple enough that subagents add overhead without benefit"
4. **Calibration check**: Asked "Did I apply appropriate skepticism?" — priming for skepticism rather than differentiation

## Changes Made

### 1. System Prompt Rewrite (Object Level)
**Before**: 480 lines with prescriptive logit thresholds, redundant sections, and 9-step workflow
**After**: 114 lines with general principles

Key deletions:
- All prescriptive logit amounts ("subtract 0.5-1.5 logits")
- Absolute thresholds ("5pp buffer", "500+ forecasters", ">$10k volume")
- Redundant sections (same concepts repeated 3-4 times)
- Prescriptive 9-step workflow
- "Explore your tools first" (wasteful token burn)
- Google Trends specific section (too narrow)
- Factor Strength table (logit scale covers this)

Key additions:
- "Hedging check" calibration principle
- "Interpretation uncertainty" principle (replaces full Definitional Questions section)

### 2. Subagent Ontology Rewrite (Object Level)
**Before**: 6 subagents (searcher, deep-researcher, estimator, quick-researcher, link-explorer, fact-checker) — 0% usage rate
**After**: 3 subagents (researcher, analyst, premortem)

| Agent | Model | Tools | Purpose |
|-------|-------|-------|---------|
| researcher | haiku | search_exa, search_news, wikipedia, search_arxiv, search_metaculus, get_metaculus_questions, polymarket_price, manifold_price, WebSearch, WebFetch | Information gathering |
| analyst | inherit | execute_code, install_package, get_cp_history, stock_price, stock_history, fred_series, fred_search, company_financials, google_trends, polymarket_history, manifold_history | Quantitative analysis |
| premortem | inherit | (same as researcher) | Devil's advocate — argues against preliminary forecast |

**New tools exposed to subagents** (previously only available to main agent):
- search_arxiv, fred_series, fred_search, company_financials
- stock_price, stock_history, google_trends (all 3 variants)
- get_cp_history, polymarket_history, manifold_history

**DRY subagent prompt**: Subagent descriptions in the system prompt are now generated dynamically from `AgentDefinition.description` fields, not hardcoded in two places.

### 3. CP History Fix (Object Level)
Switched `get_cp_history` from `/api/posts/{id}/` (returns null aggregations for resolved questions) to `/api/aggregation_explorer/?post_id={id}&aggregation_methods=recency_weighted` which works for both open and resolved questions. Added `AggregationMethod`/`AggregationHistoryPoint` Pydantic models and typed `CPHistoryResponse`/`CPHistoryEntry` output models.

### 4. Batch Resolution Fetching (Infrastructure)
Refactored resolution_update.py from 93 individual API calls to batch listing (~3 paginated calls per tournament) + targeted individual fetches only for confirmed resolutions.

### 5. Version Bump
AGENT_VERSION 0.5.0 → 0.6.0 (minor: prompt + subagent changes)
